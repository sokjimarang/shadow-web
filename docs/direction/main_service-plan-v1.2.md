# Shadow: 위임의 시작

## 해커톤 기획안 v1.2

---

## 1. 문제 정의

### 1.1 배경: AI 에이전트 시대의 도래

AI 에이전트가 인간의 업무를 대신하는 미래가 다가오고 있다. 그러나 현재와 그 미래 사이에는 분명한 간극이 존재한다.

**현재**: AI는 "도구"다. 내가 질문하면 답해준다. 주도권은 100% 인간에게 있다.

**미래**: AI는 "동료"다. 목표만 주면 알아서 실행한다. 인간은 감독하고 의사결정한다.

기술은 빠르게 발전하고 있다. 그러나 막상 AI에게 일을 맡기려 하면 사람들은 멈춘다.

**"그래서 뭘 시키지?"**

### 1.2 핵심 병목: 위임 능력의 부재

AI에게 일을 맡기는 과정은 크게 세 단계로 나눌 수 있다.

### 1단계: 발견 — "무엇을 위임할 것인가"

| 세부 단계 | 질문 | 난이도 |
| --- | --- | --- |
| 업무 인식 | "나 지금 뭐하고 있지?" | 중 |
| 업무 분해 | "이 일은 어떤 단계들로 구성되지?" | 중 |
| 위임 판단 | "이 중에 AI가 할 수 있는 게 뭐지?" | 중 |

### 2단계: 정의 — "어떻게 위임할 것인가" ⚠️ 핵심 병목

| 세부 단계 | 질문 | 난이도 |
| --- | --- | --- |
| 목표 정의 | "이 일의 성공이 뭐야?" | 중 |
| 절차 설명 | "어떤 순서로 해?" | 중 |
| **판단 기준** | **"선택의 순간에 뭘 기준으로 해?"** | **최상** |
| 예외 처리 | "이상한 상황엔 어떻게 해?" | 상 |
| **품질 기준** | **"'잘했다'의 기준이 뭐야?"** | **최상** |

### 3단계: 실행 — "위임 후 관리"

| 세부 단계 | 질문 | 난이도 |
| --- | --- | --- |
| 맥락 전달 | "필요한 정보와 기준을 넘기기" | 상 |
| 실행 위임 | AI가 실행 | 하 |
| 검증 | 결과 확인 및 피드백 | 중 |

**가장 큰 병목은 2단계 "정의", 그 중에서도 "판단 기준"과 "품질 기준"이다.**

이것들은 머릿속에만 있고, 본인도 모르는 경우가 많다.

### 1.3 왜 정의가 어려운가: 실제 예시

**"미팅 요약 정리해줘"** 라고 AI에게 시킨다고 가정하자.

절차는 간단하다: 회의록을 보고 → 핵심을 뽑고 → 정리한다.

그런데 **판단 기준**은?

- "핵심"이 뭔가? 결정사항? 논쟁점? 액션아이템?
- 누구 관점에서의 핵심인가? 대표? 실무자?
- 어느 정도 길이로 정리해야 하나?

**품질 기준**은?

- "잘 정리됐다"의 기준이 뭔가?
- 사람마다, 상황마다 다르다.

이걸 매번 말로 설명해야 하는데, 본인도 정확히 모르니까 설명을 못 한다.

### 1.4 문제의 본질

> "나도 내가 어떻게 일하는지 모른다"
> 
- 경험으로 체화된 지식은 말로 설명하기 어렵다
- 설명하려면 먼저 자기 업무를 관찰하고 분석해야 한다
- 그러나 그럴 시간도 없고, 방법도 모른다

**이것이 AI 에이전트 시대로 넘어가는 가장 큰 병목이다.**

### 1.5 기존 접근법의 한계

| 접근법 | 방식 | 한계 |
| --- | --- | --- |
| AI 어시스턴트 | 사용자가 먼저 요청 | 뭘 요청할지 모름 |
| RPA/Task Mining | 시스템 이벤트 로그 분석 | 행동의 의미/맥락 파악 못함 |
| 프로세스 컨설팅 | 인터뷰 + 문서화 | 비용 높음, 암묵지 누락 |

**공통된 한계**: 사용자가 자신의 업무를 "설명"해야 한다는 전제에서 출발한다.

## 2. 솔루션: Shadow

### 2.1 핵심 인사이트

> "정의가 어려우면, 관찰하면 된다."
> 

말로 설명하는 대신, 그냥 평소처럼 일하면 된다. AI가 보고 배운다.

**기존 방식:**
사용자가 머릿속 지식을 → 말로 설명 → AI가 이해
(병목: "설명" 단계)

**Shadow 방식:**
사용자가 그냥 일함 → AI가 화면 관찰 → 패턴 추출 → 사용자에게 확인 → 명세서 생성
(병목 우회: 설명 대신 관찰)

### 2.2 제품 정의

**Shadow**는 사용자의 반복적인 업무 패턴을 관찰하여 **에이전트 명세서**를 자동 생성하는 AI다.

**에이전트 명세서**란:

- AI 에이전트가 특정 업무를 수행하기 위해 필요한 모든 정보를 담은 문서
- 업무의 절차, 판단 기준, 예외 처리, 품질 기준을 포함
- 이 명세서가 있으면 AI 에이전트는 해당 업무를 대신 수행할 수 있다

**Shadow가 하는 일:**

1. 그림자처럼 조용히 사용자의 업무를 관찰
2. 반복되는 패턴을 발견
3. 패턴에서 규칙과 기준을 추론
4. 불확실한 부분은 사용자에게 직접 확인
5. 확인된 내용을 바탕으로 에이전트 명세서 생성 및 업데이트

### 2.3 핵심 가치 제안

| 기존 | Shadow |
| --- | --- |
| "어떻게 일하세요?" (열린 질문) | "이렇게 하신 것 같은데 맞나요?" (닫힌 질문) |
| 사용자가 처음부터 설명 | AI가 먼저 관찰하고 가설 제시 |
| 암묵지 끌어내기 어려움 | 구체적 상황에서 확인하므로 쉬움 |
| 일회성 문서화 | 지속적으로 업데이트되는 Living Document |

### 2.4 Human-in-the-Loop (HITL)

Shadow는 완전 자동화를 목표로 하지 않는다. 대신 **사람과 AI의 협업**을 통해 정확한 명세서를 만든다.

**HITL이 필요한 이유:**

- 같은 행동도 상황에 따라 다른 의미를 가질 수 있다
- AI의 추론은 틀릴 수 있다
- 사용자만 아는 맥락이 있다

**HITL 방식:**

1. AI가 패턴을 발견하고 규칙을 추론
2. 불확실한 부분에 대해 사용자에게 질문
3. 사용자는 간단히 확인하거나 수정
4. 확인된 내용이 명세서에 반영

**핵심 전환:**

- ❌ "이 작업의 판단 기준이 뭔가요?" (열린 질문)
- ✅ "금액이 100만원 이상이면 노란색으로 표시하시는 것 같은데, 맞나요?" (닫힌 질문)

**HITL 질문 유형:**

| 유형 | 트리거 | 질문 예시 |
| --- | --- | --- |
| 이상 탐지 | 평소 패턴과 다른 행동 | "보통은 A→B→C인데, 이번엔 A→D→C 하셨네요. 특별한 이유가 있나요?" |
| 대안 제시 | 더 효율적인 방법 존재 | "Ctrl+C 대신 우클릭 복사를 쓰시는 특별한 이유가 있나요?" |
| 가설 검증 | 패턴에서 규칙 추론 | "금액이 100만원 넘으면 상사 확인을 받으시는 것 같은데, 맞나요?" |
| 품질 확인 | 최종 산출물 분석 | "결과물에 항상 날짜를 넣으시는데, 필수 요소인가요?" |

### 2.5 산출물: 에이전트 명세서

Shadow의 최종 산출물은 **에이전트 명세서**다.

이 명세서는:

- 특정 반복 업무에 대한 완전한 정의
- AI 에이전트가 해당 업무를 수행하는 데 필요한 모든 정보 포함
- 사용자의 피드백을 반영하여 지속적으로 개선

명세서에 포함되는 내용:

- **트리거**: 언제 이 업무가 시작되는가
- **워크플로우**: 어떤 단계로 진행되는가
- **판단 기준**: 선택이 필요한 순간, 무엇을 기준으로 하는가
- **예외 처리**: 예상과 다른 상황에서 어떻게 하는가
- **품질 기준**: 결과물이 갖춰야 할 조건은 무엇인가

## 3. 전체 플로우

### 3.1 5단계 순환 구조

Shadow는 다음 5단계를 반복하며 명세서를 점진적으로 완성한다.

```
OBSERVE → ANALYZE → CLARIFY → PROCESS → NOTIFY → (반복)
(관찰)    (분석)    (확인)    (처리)    (알림)

```

### 3.2 단계별 설명

### 1. OBSERVE (관찰)

사용자 컴퓨터에서 백그라운드로 실행되며 업무 데이터를 수집한다.

수집 대상:

- 화면 변화 (Before/After 스크린샷)
- 마우스 이벤트 (클릭 위치, 타입)
- 키보드 이벤트 (입력 내용, 단축키)
- 활성 윈도우 정보 (어떤 앱에서 작업 중인지)

### 2. ANALYZE (분석)

수집된 데이터를 AI가 분석하여 행동을 이해하고 패턴을 감지한다.

행동 이해:

- Before/After 스크린샷 + 이벤트 로그를 결합하여 분석
- 각 행동에 의미 부여 (예: "Gmail 검색창에 주문번호 입력")

패턴 감지:

- 여러 세션의 행동 시퀀스를 비교
- 반복되는 패턴 식별
- 패턴 내 변형 분석 (예: "보통은 C열인데, 가끔 D열에 입력")

### 3. CLARIFY (확인) — HITL

불확실한 부분에 대해 Slack을 통해 사용자에게 확인한다.

### 4. PROCESS (처리)

사용자 응답을 해석하고 명세서를 업데이트한다.

- 버튼 응답: 직접 반영
- 텍스트 응답: AI가 의도를 파악하여 구조화
- 명세서 버전 관리 및 변경 이력 기록

### 5. NOTIFY (알림)

업데이트 완료를 사용자에게 알린다.

- Slack으로 변경 사항 요약 전송
- 상세 내용은 웹 인터페이스에서 확인 가능

### 3.3 콜드스타트 해결

패턴으로 인정하기 위한 최소 조건: **동일/유사 패턴 3회 이상 관찰**

- 1회 관찰: 아직 패턴인지 알 수 없음
- 2회 관찰: 유사성 감지, 확신 낮음
- 3회 관찰: 패턴 확정, 명세서 초안 생성 가능

## 4. 기술 아키텍처

### 4.1 시스템 구성

Shadow는 세 가지 주요 컴포넌트로 구성된다.

**1. Shadow Agent (사용자 컴퓨터)**

- CLI로 실행
- 화면 캡처 및 입력 이벤트 수집
- 수집된 데이터를 서버로 전송

**2. Shadow Server**

- Vision LLM을 활용한 행동 분석
- 패턴 감지 및 명세서 관리
- HITL 질문 생성 및 응답 처리

**3. 사용자 인터페이스**

- Slack Bot: HITL 대화, 상태 확인, 명령어 처리
- Web UI: 명세서 상세 보기, 변경 이력 확인

### 4.2 기술 스택

| 구분 | 기술 | 비고 |
| --- | --- | --- |
| 화면 캡처 | mss | Python 라이브러리 |
| 입력 로깅 | pynput | 마우스/키보드 이벤트 |
| Vision LLM | Claude Opus 4.5 | Primary |
| Vision LLM | NVIDIA NIM (Qwen 3 VL) | Alternative |
| Slack 연동 | Slack Bolt | Python SDK |
| Web UI | Streamlit | 빠른 프로토타이핑 |

### 4.3 모델 교체 가능 설계

Vision LLM은 추상화 레이어를 통해 교체 가능하도록 설계한다.

- Primary: Claude Opus 4.5
- Alternative: NVIDIA NIM (Qwen 3 VL)

## 5. 사용자 인터페이스

### 5.1 CLI (Shadow Agent)

Shadow는 터미널에서 실행된다.

주요 명령어:

- `shadow start`: 관찰 시작
- `shadow stop`: 관찰 중지
- `shadow status`: 현재 상태 확인
- `shadow specs list`: 명세서 목록 조회

### 5.2 Slack Bot

사용자와의 주요 소통 채널이다.

지원 기능:

- HITL 질문 수신 및 응답
- 상태 확인 ("현재 상태")
- 패턴 목록 조회 ("내 패턴 보여줘")
- 녹화 제어 ("녹화 중지")
- 명세서 수정 요청 ("이 조건 50만원으로 바꿔줘")
- 업데이트 알림 수신

### 5.3 Web Interface

명세서의 상세 내용을 확인하는 용도다.

제공 기능:

- 명세서 상세 보기
- 변경 전/후 비교 (Diff 뷰)
- 변경 이력 조회

- 언제든 수집된 데이터 삭제 가능
- 명세서 내용 투명하게 공개

## 7. 성공 지표

### 7.1 핵심 데모 목표

> "3회 관찰 후, 사용자가 2개 질문에 답하면 spec이 생성되고, 다음 회차부터는 같은 판단을 질문 없이 재현한다 (동일 판단 기준을 다시 묻지 않음)."
> 

### 7.2 세부 성공 기준

| 기준 | 설명 |
| --- | --- |
| **패턴 발견** | 3회 반복 업무에서 패턴을 자동으로 발견 |
| **HITL 동작** | 질문-응답 사이클이 명세서에 반영됨 |
| **E2E 플로우** | 관찰 → 분석 → 질문 → 응답 → 명세서 생성의 전체 사이클 시연 |

## 9. 핵심 메시지

### 9.1 문제-솔루션 연결

**문제**: AI에게 일을 맡기고 싶지만, 내 업무를 어떻게 설명해야 할지 모르겠다.

**인사이트**: 설명이 어려우면, 관찰하면 된다.

**솔루션**: Shadow가 업무를 관찰하고, 패턴을 발견하고, 확인을 거쳐 에이전트 명세서를 만든다.

### 9.2 한 줄 요약

> "AI에게 일을 시키는 첫 단계는, AI가 당신의 일을 이해하는 것입니다.말로 설명하지 마세요. 그냥 일하세요. Shadow가 보고 배웁니다."
